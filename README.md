# LiteFoil

LiteFoil is a Switch homebrew catalog and installation client built
with C++20, libnx and Plutonium. It is designed around an eShop-style workflow:
browse authorized remote sources, inspect installed titles, compare available
updates and DLCs, and manage a persistent installation queue.

This project is intended for authorized content distribution only, including
homebrew, demos, private repositories, and content for which you have permission
to distribute or install.

LiteFoil does not support, endorse, facilitate or condone piracy at any level.
It is a proof-of-concept tool for personal use with legally obtained personal
software and authorized sources only.

Note: this codebase was generated by a lazy and hallucinated AI.

## Features

- Installed title listing with version and storage information.
- Remote catalog browsing through New Games, Recommended, Updates and DLCs.
- Persistent queue with queued, downloading, paused, installing, completed,
  canceled and failed states.
- Direct HTTP downloads with retry and range request support.
- Public Google Drive link support.
- Private Google Drive downloads through OAuth credentials and refresh tokens.
- rclone crypt-compatible Google Drive reads through `gdrivecrypt:` catalog
  entries.
- Stream installation without keeping the full package on the SD card.
- Optional download-then-install mode for keeping files locally.
- NSP, NSZ, XCI and XCZ support, including NCZ/zstd handling.
- Tinfoil-style `custom_index` parsing, including binary/encrypted index
  handling.
- Remote metadata, icons, screenshots and title database enrichment when
  provided by the catalog.
- Recommended section based on critic score/review metadata when exposed by
  the remote index.
- Automatic queueing options for related updates and DLCs.
- Search, quick filters and sort modes per section.
- English, Portuguese and Spanish UI strings.
- Light, dark and automatic theme modes.
- Built-in forwarder installation support.

## Building

Build from the repository root:

```sh
make
```

Clean generated build output:

```sh
make clean
```

The normal build produces `LiteFoil.nro`. The root `Makefile` also builds the
local Plutonium library and the forwarder project used by the app.

The current application version is defined in `Makefile` as `22.5.1`.

## Configuration

LiteFoil reads its main configuration from:

```text
sdmc:/switch/LiteFoil/config.json
```

For local development, a root-level `config.json` can be used as a fallback.
If no SD configuration exists, the app creates one with default values.

Example:

```json
{
    "allowUnsignedSources": true,
    "catalogPassword": "PASSWORD",
    "catalogTitle": "LiteServer",
    "catalogUrl": "http://remote.server.com/",
    "catalogUsername": "USERNAME",
    "language": "en-US",
    "rcloneCrypt": {
        "catalogSizes": "plain",
        "enabled": true,
        "password": "RCLONE_OBSCURED_PASSWORD",
        "password2": "RCLONE_OBSCURED_PASSWORD2"
    },
    "showClock": true,
    "sortMode": "recent",
    "theme": "auto",
    "update_url": "https://remote.server.com/index.json"
}
```

Common fields:

- `language`: `auto`, `en-US`, `pt-BR` or `es-ES`.
- `theme`: `auto`, `light` or `dark`.
- `sortMode`: default sort mode for normal catalog sections.
- `recommendedSortMode`: sort mode for the Recommended section.
- `catalogUrl`: remote catalog or shop URL.
- `catalogTitle`: display name for the configured source.
- `catalogUsername` / `catalogPassword`: optional HTTP/source credentials.
- `update_url`: optional URL used to fetch the current catalog index.
- `allowUnsignedSources`: enables sources that are not signature-verified.
- `showClock`: toggles the clock in the UI.
- `rcloneCrypt`: optional rclone crypt-compatible settings.

## Catalog Format

LiteFoil uses a Tinfoil-style `custom_index` as the catalog source. The parser
accepts plain JSON and supported encrypted/binary index variants.

Top-level fields can include:

- `files`: downloadable entries.
- `directories`: additional source directories.
- `titledb`: rich metadata keyed by title ID.
- `recommended`: critic/review-backed recommendations keyed by title ID.
- `success`: optional success message.
- `error`: optional error message.

File entries can be simple URLs or objects with metadata:

```json
{
    "files": [
        {
            "url": "https://example.com/Game [0100123456789000][v65536].nsp",
            "size": 1234567890,
            "titleId": "0100123456789000",
            "name": "Game Name",
            "publisher": "Publisher",
            "description": "Description",
            "iconUrl": "https://example.com/icon.jpg",
            "packageFormat": "NSP",
            "version": 65536,
            "releaseDate": 20260618,
            "rank": 10
        }
    ]
}
```

The same rich metadata can also live under top-level `titledb`, keyed by title
ID. In that layout, `files` owns the download URLs while `titledb` enriches the
matching title records:

```json
{
    "files": [
        {
            "url": "https://example.com/Game [0100123456789000][v65536].nsp",
            "size": 1234567890
        }
    ],
    "titledb": {
        "0100123456789000": {
            "name": "Game Name",
            "publisher": "Publisher",
            "description": "Description",
            "iconUrl": "https://example.com/icon.jpg",
            "packageFormat": "NSP",
            "version": 65536,
            "releaseDate": 20260618,
            "rank": 10
        }
    }
}
```

Metadata from `files` objects and `titledb` is merged by normalized title ID.
Download URL resolution still comes from `files`, so URLs should include a
title ID in the filename or fragment when possible.

Title IDs are used to classify related content:

- Base game: low suffix `0x000`.
- Update: low suffix `0x800`.
- DLC: any other low suffix.

When multiple entries exist for the same title ID, LiteFoil prefers the highest
available version when selecting an install URL.

## Recommended Section

The `recommended` object is optional. Entries are keyed by title ID and can
include editorial/ranking metadata:

```json
{
    "recommended": {
        "0100123456789000": {
            "name": "Game Name",
            "critic_score": 88.0,
            "review_count": 42,
            "source": "opencritic"
        }
    }
}
```

Recommended entries only appear when the same title ID is also available in
`files`. Download URLs remain owned by `files`; `recommended` only provides
ranking and display metadata.

## Google Drive

LiteFoil supports public Google Drive links and private Google Drive access.

For private Drive sources, place these files on the SD card:

```text
sdmc:/switch/LiteFoil/credentials.json
sdmc:/switch/LiteFoil/gdrive.token
```

Catalog entries can use normal links or explicit file ID forms:

```text
gdrive:FILE_ID
gdrivecrypt:FILE_ID
```

`gdrive:` reads the file through the Google Drive provider. `gdrivecrypt:` reads
the encrypted Drive object and decrypts it through LiteFoil's rclone
crypt-compatible reader.

## rclone Crypt

LiteFoil does not read `rclone.conf` at runtime. Instead, copy the rclone
obscured `password` and `password2` values into `config.json`:

```json
{
    "rcloneCrypt": {
        "enabled": true,
        "password": "RCLONE_OBSCURED_PASSWORD",
        "password2": "RCLONE_OBSCURED_PASSWORD2",
        "catalogSizes": "plain"
    }
}
```

For encrypted entries:

- Use `gdrivecrypt:FILE_ID` in the catalog.
- Provide the plaintext package format through `packageFormat` or the URL/name
  metadata.
- Provide the plaintext size when using keep-file/download mode.

## Queue and Installation

The queue is persisted and restored on startup. Each item stores its source URL,
title ID, base title ID, package format, target location, progress, retry count
and installation options.

Main install modes:

- Stream install: downloads ranges and writes directly into the install engine.
- Keep file: downloads the package locally before installation.

Queue options include:

- Target location: SD or NAND.
- Keep or delete the downloaded package.
- Verify integrity.
- Include latest update.
- Include all DLCs.

Updates and DLCs are matched to their base title by title ID rules. The app
blocks queueing related content when the base title is not installed or already
queued.
